02 Problem · 它解决什么问题

它解决什么问题？

Swarm 解决的不是"如何做一个完整生产级 Agent 平台"，而是"多 Agent 协作最小闭环到底长什么样"。

在 Agent 系统里，常见问题包括：

  1. 一个 Agent 什么时候该调用工具？
  2. 工具调用结果怎么回到大模型？
  3. 多个 Agent 之间怎么交接任务？
  4. 一轮任务执行到什么时候应该停止？
  5. 对话历史和业务上下文分别放在哪里？

Swarm 用非常少的抽象把这些问题串起来：

Swarm
负责调度。整个 Agent Loop 的执行入口。
Agent
定义角色、模型、指令和工具清单。
Response
承载一次完整 run 后的整体结果。
Result
封装单个工具函数的返回值、上下文变化与可能的 Handoff。

所以它的价值不是"大而全"，而是把 Agent 编排的骨架暴露得很清楚。

03 Plain Language · 讲给不懂的人

用最小例子讲给完全不懂的人

可以把 Swarm 想成一个小型任务调度台。

用户提出任务后，调度台先叫来一个 Agent。这个 Agent 可以直接回答，也可以调用工具。如果工具执行后发现任务该交给另一个 Agent，工具就返回一个新的 Agent 对象。Swarm 接住这个对象，把当前负责人切换过去，然后让新 Agent 继续处理。

04 Concrete · 状态推导

一次 run() 的状态变化

Swarm 没有复杂数学骨架，但可以用一次运行的状态变化来理解。

假设一次 run() 初始状态是：

active_agent = SalesAgent
history = [用户消息]
context_variables = {"customer_type": "enterprise"}
max_turns = 5

运行过程中可能发生：

  1. SalesAgent 认为需要查询报价，返回 tool_calls。
  2. Swarm 根据函数名找到真实 Python 函数并执行。
  3. 函数返回 Result(value="报价已生成", context_variables={"quote_id": "Q001"})。
  4. Swarm 把工具结果追加到 history。
  5. Swarm 把上下文更新为 {"customer_type": "enterprise", "quote_id": "Q001"}。
  6. 如果函数还返回 SupportAgent，则 active_agent 切换为 SupportAgent。
  7. 下一轮循环继续，直到 LLM 不再产生工具调用。
关键点
  • history 主要给 LLM 看 — 记录"说了什么、做了什么"。
  • context_variables 主要给工具和代码看 — 记录"关键业务状态变成什么"。
05 Analogy · 类比与边界

类比（含边界）

好类比能让人秒懂，坏类比会埋下错误的直觉——所以两边都要写清楚。

类比：前台调度员 + 专家小组。
准确的地方
  • 前台负责把任务派给合适的人。
  • 每个专家有自己的身份、说明书和能力清单。
  • 专家可以做事，也可以把任务转交给另一个专家。
  • 任务过程会留下记录，方便下一位专家接着处理。
会误导的地方
  • Swarm 里的 Agent 不是独立进程，更像是一组带不同 instructions 和 tools 的配置对象。
  • Handoff 不是复杂工作流引擎，本质只是某个工具函数返回了一个新的 Agent。
  • Swarm 不自动判断"用户目标是否真正完成"，它主要根据是否还有工具调用决定循环是否继续。

07 Scenarios · 适用场景

出现在哪些模型 / 场景

Swarm 是 OpenAI 发布过的一个轻量级多 Agent 编排示例 / 框架，适合用来学习 Agent 系统的基础结构。

典型场景
  • 学习多 Agent Handoff 的最小实现。
  • 理解 Function Calling 在 Agent Loop 里的位置。
  • 搭建客服、销售、技术支持等角色分工 Demo。
  • 快速验证"一个 Agent 调工具后交给另一个 Agent"的流程。
  • 作为理解更复杂框架的前置样板。
不适合直接承担
  • 需要强状态机控制的复杂工作流。
  • 需要长期记忆、断点恢复、权限隔离的生产系统。
  • 需要严格结果验证、失败重试和审计日志的高可靠业务。
  • 需要可视化编排和复杂并发调度的 Agent 平台。

09 Misconceptions · 我的误解

我曾经的误解（如果有）

把错的写出来比把对的写出来更重要——错的更暴露盲区。

误解 · i

Swarm 是一个完整的 Agent 产品平台。

修正日期 · 2026·05·24
修正

Swarm 更像是一个教学价值很高的极简多 Agent 编排框架。它展示了 Agent、工具调用、上下文更新、Handoff 和循环退出的核心闭环，但没有内置长期记忆、显式目标验证、复杂状态机和生产级安全控制。

误解 · ii

Agent 之间的交接需要一个复杂的调度算法。

修正日期 · 2026·05·24
修正

在 Swarm 里，Handoff 的本质很简单：一个工具函数返回新的 Agent 对象，Swarm 把 active_agent 指向它，下一轮循环就由新 Agent 继续。

10 Disclosure · 诚实附注

诚实附注

这篇笔记的边界、引用和复用声明。

Honest Notes

这份笔记是基于我的原始学习笔记「Agent 开源项目学习 一 Swarm」改写成"全球热门 Agent 产品 / 项目学习"格式。

Swarm 在产品学习里的价值主要是"理解 Agent 编排原型"，不是评估一个成熟商业产品。若要做产品选型，需要进一步对比 LangGraph、AutoGen、CrewAI、OpenAI Agents SDK 等框架在状态管理、可观测性、部署、安全和生态上的差异。

此外，Agent 框架变化很快。公开写作或选型前，需要重新核对 Swarm 仓库状态、OpenAI Agents SDK 的更新，以及社区是否仍建议在新项目中使用 Swarm。
